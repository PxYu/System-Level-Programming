CLR Profiler Hotspots Tracker (profilerhst) Sample 

 

Sample Overview

This sample illustrates how to use the CLR Profiler API for identifying hotspots in managed code. To accomplish this, the profiler monitors the following CLR events:

 

·         Function Enter-Leave events

·         Code Transition events

·         Exception events

·         Suspend-Resume events

·         Threads birth-death events

 

Under the sheets, the profiler maintains a list of threads. Each thread maintains a list of invoked functions. The profiler also maintains a set of timers for each thread and function. One timer is used to compute the total time that thread was alive. The suspension timer is used to compute the total time the thread was suspended. A third timer computes the overhead caused to the thread by the profiling process. This time is subtracted from the total time for the thread.

 

The output generated by the profiler is a semicolon delimited file of data. This can be imported in Microsoft Excel and then analyzed. To modify the format of the output, see any of the Dump() methods in ProfilerInfo.cpp. If a failure occurs during the program’s execution, the profiler will disable the reception of profiling events and print the error to the standard output. The profiler will not attempt to terminate the execution of the managed program.

   

We strongly recommend reviewing the Profiling API ReadMe items before attempting any changes to the tool since the profiler API has by default very loose error checks in order to improve performance. This means that the CLR can crash if care is not used when passing null arguments to various Profiler API methods.

 

 

Building the Sample

To build the ProfilerHST sample you will need to have a fairly recent version of the Win32 SDK installed on your system. The sample was intended to be built (via nmake) from the version of the C++ compiler that comes with the .NET SDK. Moreover, the sample is dependent on various CLR headers/libs. Below is the list of files included for the sample.

  

profiler\hst

readme.htm

profilerhst.mak

profilerhst.cpp

profilerhst.def

profilerinfo.h

profilerInfo.cpp

enableprofiler.bat

profilercallback.h  

profilercallback.cpp

 

profiler\include

regutil.h

basehdr.h

avlnode.h

basehlp.h

regutil.hpp

dllmain.hpp

avlnode.hpp

basehlp.hpp

prf_common.h

container.hpp

classfactory.hpp

 

  

·         Ensure that the environment (command shell) is setup to build using nmake. The makefile assumes that the location of the CLR and Win32 headers/libraries are included in the Win32 environment variables INCLUDE and LIB. The .NET SDK setup will modify these environment variables to include the location for the CLR headers/libraries.

 

·         By default, the makefile assumes that the Win32 headers/libs can be found from the environment variables

 

LIB

INCLUDE

 

If necessary, modify these two environment variables to reflect where the Win32 headers/libs are located on your system.

 

·         By default, the makefile also assumes that the CLR headers/libs can be found from the environment variables

 

LIB

INCLUDE

 

Again, if necessary, modify these two environment variables to reflect where the CLR headers/libs are located on your system.

 

·         From the directory containing the makefile, invoke

 

nmake –f profilerhst.mak

 

to build the sample. The end result should be a debug version of profilerhst.dll. The makefile can be modified to build a retail version if desired.

 

 

Running the Sample

To enable the profiler, run the batch file enableprofiler.bat. This will register the profiler in the registry and setup an environment for using the profiler. Running any managed program under the environment will then be profiled.
